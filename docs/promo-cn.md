## 朋友圈短版（推荐）

最近做了一个我自己一直想要的工具：**UItoPrompt**。

我们经常看到一个喜欢的网页或截图，然后丢给 AI 说“照这个风格做一个”。但“极简、现代、有呼吸感”这类形容词，通常不足以让模型真正复现它的排版、色彩、留白和组件节奏。

UItoPrompt 做的是中间最缺的那一层：把网页或图片拆成可检查的视觉证据，再整理为 **StyleSpec + 15 章 Systematic Prompt + CSS 变量 + 可复用 Skill**。它会明确区分哪些是直接观察、程序测量、系统推断、视觉转译，以及哪些仍然未知。

如果输入只是一张截图，它不会假装知道原网页的 DOM、字体文件、移动端断点、Hover 状态或动效；证据不足的部分会保留为未知，并告诉后续模型该如何保守处理。

默认模式只提炼视觉语言，不复制原品牌、文案、Logo、图片和受保护资产。你可以把生成的 Skill 直接交给 Codex、Claude Code 或 Cursor，让不同模型长期复用同一套设计规则。

第一版已完成真实 URL 取证、本地图片像素分析、Prompt 编译、Skill 导出和桌面/手机端验证。审核通过后会开源，并部署到 **uitoprompt.com**。

如果你也常常想告诉 AI：“我想要的不是这个网页的内容，而是它背后的设计逻辑”，这个工具就是为你做的。

## 推荐正文

最近做了一个我自己一直很想要的工具：**UItoPrompt**。

举个很具体的例子。

你看到一个网页，喜欢它的排版、颜色、留白和组件节奏。以前通常是截张图，丢给大模型说：“照这个风格做一个。”模型也许会写出一段听起来很专业的话——极简、现代、克制、富有呼吸感——但真正开始生成时，结果往往还是熟悉的紫色渐变、三栏卡片和大圆角。因为“形容得像”，不等于“规则提取得对”。

UItoPrompt 想解决的就是这中间缺失的一层：**把一个网页或一张图片，转换成大模型真正可以执行、检查和长期复用的设计规则。**

使用方式很直接：粘贴一个公开网页，或者上传一张界面截图。UItoPrompt 会把参考内容拆成颜色角色、字体层级、间距节奏、网格与布局、圆角和阴影、组件语法、内容密度、响应式规则、交互与动效，以及明确的负面约束。最后输出的不是一段“氛围描述”，而是一套完整的 Systematic Prompt、StyleSpec、CSS 变量、证据报告和可直接交给 Codex、Claude Code、Cursor 等工具使用的 Skill。

它和普通“截图转 Prompt”工具最大的区别，是会把结论分清楚：

- 哪些是画面里直接看到的；
- 哪些是浏览器或程序实际测量出来的；
- 哪些只是基于证据做出的推断；
- 哪些是从海报、照片或插画转译成的 UI 规则；
- 哪些来自用户确认；
- 哪些信息目前根本不知道。

比如只有一张桌面截图时，工具不会假装自己知道原网页的 DOM、真实字体文件、移动端断点、Hover 状态和完整动效。这些内容会被明确标成“未知”，并告诉后续模型应该保留占位、采用什么保守方案，以及还需要补充什么证据。对我来说，这种诚实比一个虚假的“98% 还原度”更重要。

它适合的场景很多：

做新产品时，可以把喜欢的网页整理成一套不依赖原品牌的视觉语言；重构旧项目时，可以把现有界面提炼成团队共享的设计规则；设计师可以把参考图交给工程 Agent，而不必每次重新解释审美；开发者可以把同一套 Style Skill 放进多个项目，让不同模型的输出更稳定；看到一本杂志、一张海报或一组摄影，也可以只提取它的节奏、对比、色彩和形状，把这些特征转译成原创的 Web UI，而不是复制原作品。

它内部的工作流程是：

**网页 / 图片 → 视觉证据 → StyleSpec → Systematic Prompt → 可复用 Skill**

网页模式会尽量读取真实的结构、计算样式、字体、资源和视口信息；图片模式会分析像素、颜色、构图和可见层级。两种来源最后进入同一个结构化规范，再由确定性的编译器生成 15 个固定章节的 Prompt，包括目标范围、权利边界、证据优先级、视觉原则、设计变量、组件语法、响应式、交互动效、无障碍、禁止事项、未知处理、验收清单和迭代方式。

项目默认使用“提炼风格”模式：保留视觉语法，但移除原品牌、文案、Logo、图片、专有图标和受保护资产。只有在页面属于自己或已经获得授权时，才使用更忠实的“授权重建”模式。

现在第一版已经完成本地产品、核心 Skill、图片分析、URL 安全取证、Systematic Prompt 编译器、样例项目和完整导出。桌面端与手机端都做了真实浏览器验收；StyleSpec、Prompt 章节、下载文件、键盘操作、缩减动效和安全边界都有自动测试。
